# Changelog

## 0.2.9

변경 피드 삭제 — 3가지.

- **전체 삭제**: 상단 버튼으로 피드 싹 비우기.
- **선택 삭제**: "선택" 모드 → 항목 클릭/체크로 여러 개 고른 뒤 "선택 삭제 (N)".
- **한개씩 삭제**: 각 항목 시간 옆 `×` 클릭.
- 지운 항목은 provider 히스토리에서도 빠져 **재오픈해도 안 돌아온다**.

## 0.2.8

청중 라우팅 + 위험도 표시 + 토스트 정리.

- **청중 라우팅**: 저장·커밋은 아직 로컬(공유 전)이라 **작성자 본인에게만**(자기점검),
  푸시·PR 은 공유된 순간이라 **팀 전체**에. "공유 안 된 코드엔 남이 영향받을 수 없다."
  공유분(push/pr)만 히스토리에 남겨 늦게 온 팀원에게 백필.
- **위험도 뱃지**: 🔴 높음(계약 깨짐·부서질 수 있음) / 🟡 주의(추가) / ⚪ 참고(내부) + 영향 N곳.
- **토스트는 위험(high)만**: 오른쪽 아래 알람은 high 가 내 코드에 닿을 때만 — 저위험 팝업 소음 제거.

## 0.2.6

윈도우에서 두뇌에 **영영 연결 안 되던**("Ripple X" 빨간 상태) 진짜 원인 수정.

- **서버를 IPv4(`0.0.0.0`)에 명시 바인딩.** host 를 생략하면 Node 는 IPv6 `::` 에 묶는데,
  맥은 듀얼스택이라 127.0.0.1(IPv4)도 우연히 통과되지만 **윈도우는 그 소켓이 V6ONLY 라
  IPv4 연결을 거부**한다. 발견(udp4)·헬스체크·클라이언트가 전부 IPv4인데 서버만 IPv6라
  윈도우에서만 어긋나 무한 재연결이 돌았다. 서버도 IPv4로 통일해 해결.
  (`RIPPLE_HOST=127.0.0.1` 로 두면 LAN 노출 없이 로컬 전용.)
- 클라이언트·기본 설정·헬스체크를 모두 `127.0.0.1` 로 정렬(0.2.x 누적).
- 두뇌 자동기동 시 **stderr 캡처** → 또 막히면 진짜 원인이 "Ripple" 출력 채널에 찍힌다.
- 끊긴 동안의 저장 변경을 큐에 보관했다 재연결 시 전송(유실 방지).

## 0.2.5

변경 출처를 피드 뱃지로 구분 — **저장 / 커밋 / 푸시 / PR**.

- **커밋 감지**: `.git/logs/HEAD` reflog 감시 → 로컬 커밋이면 그 커밋의 영향을 분석(초록 뱃지).
  체크아웃·리셋·머지는 제외(reflog 액션이 `commit` 인 것만).
- **푸시 감지**: 원격 추적 ref reflog 의 `update by push` 액션으로 푸시를 잡아(주황 뱃지)
  fetch/pull 과 구분. 같은 sha 는 한 번만 분석(dedup).
- 커밋/푸시/PR 은 피드에만(조용히) — 능동 팝업은 라이브 저장만.
- 프로토콜에 `ChangeSource`(save/commit/push/pr)·`CommitRef` 추가. 통합 테스트로 고정.

## 0.2.4

Claude Code·외부 도구의 **원자적 저장**(임시파일+rename)도 변경 감지.

- 원자적 저장은 inode 가 바뀌어 VS Code 가 onDidChange 가 아니라 onDidCreate 로 보고한다 →
  기존엔 인덱스만 갱신하고 변경을 안 보냈다(편집해도 0건). create 도 diff 경로로 보내 해결.
- 삭제 핸들러에 stat 가드(원자적 rename 의 '가짜 삭제'로 기준선 안 날리게).

## 0.2.3

큰 모노레포가 **연결은 되는데 영원히 먹통**이던 치명 버그 수정.

- **두뇌 인바운드 상한 512KB → 32MB.** register 는 전체 인덱스(수천 파일의 exports/imports/refs)를
  동봉하는데, 큰 repo(예: 3000파일 모노레포)는 인덱스가 512KB 를 넘겨 register 프레임이 거부됐다.
  거부되면 무한 재연결만 돌고 그 창은 변경을 영영 못 보냈다 — "설치했는데 안 됨"의 진짜 원인.
  두뇌 로그에 `Max payload size exceeded` 가 반복되던 증상. 신뢰 그룹 전제라 5000파일 인덱스가
  들어갈 만큼 넉넉히 상향.
- 회귀 테스트 추가: 512KB 초과 인덱스 register 가 수락되는지 통합 테스트로 고정.

## 0.2.2

윈도우 연결 안 됨("Ripple X" 빨간 상태) 수정.

- **로컬 두뇌 주소를 127.0.0.1 로 고정** — 윈도우는 `localhost` 를 `::1`(IPv6) 로 먼저 해석해
  IPv4 로 뜬 두뇌에 `ECONNREFUSED ::1:7077` 이 나며 영구 끊김이었다(맥은 127.0.0.1 우선이라 안 보임).
  설정에 `ws://localhost:7077` 이 남아 있어도 127.0.0.1 자동기동 경로로 수렴.
- **끊긴 동안의 저장 변경 유실 방지** — 연결이 끊겨 있으면 변경을 큐에 보관했다 재연결 즉시 전송.
  (이전엔 기준선만 전진하고 메시지는 버려져, 재연결돼도 그 변경이 영영 안 보였다.)
- **자동기동 두뇌의 조용한 크래시 노출** — 종료코드를 Ripple 출력 채널에 로그. 서버는 listen 실패
  (포트 충돌 등)를 잡아 깔끔히 종료.
